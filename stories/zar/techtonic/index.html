TECHTONIC — BUILT THREE TIMES, SHIPPED ONCE

670+ components, 37 Figma pages, 103 icons, 3 platforms. A design system built, rebuilt, and rebuilt again as ZAR's product evolved beneath it.

Company ZAR Year 2024 – 2025 Role Head of Design

THE CHALLENGE

ZAR's product was built by freelancers. Multiple designers, multiple engineers, no shared system, no shared language. The Figma files were a graveyard of duplicated components, orphaned frames, and styles that varied by who designed them and when. Every screen looked slightly different.

The product spanned three platforms — iOS, Android, and web — each with its own visual inconsistencies. Engineers had no reliable source of truth for design specifications. Every feature handoff was a negotiation: "Which version of this button do I use? What's the correct spacing? Is this the right shade of gray?"

To make it harder, the product itself was evolving rapidly. ZAR pivoted its positioning multiple times, which meant the design system couldn't be a static artifact — it had to keep pace with a moving target.

670+ components across 3 platforms

THE APPROACH

I built Techtonic three times. Not because of poor planning — because the product changed underneath it. Each version was a response to a new product direction, new platform requirements, and new scale demands.

Version 1 was a cleanup: audit everything, consolidate duplicates, establish base tokens and core components. It brought order to chaos but was designed for a product scope that soon expanded.

Version 2 introduced platform-specific adaptations. iOS, Android, and web each got their own component sets that shared the same tokens but respected platform conventions. This version also introduced the icon library — 103 custom icons designed specifically for ZAR's product vocabulary.

Version 3 was the system that shipped. It integrated with GitHub and Storybook, giving engineers a coded component library that mirrored Figma 1:1. Design tokens were synced between Figma and code. 37 Figma pages organized by component category, pattern, and platform. Full documentation for every component.

THE SOLUTION

Techtonic in its final form was a comprehensive multi-platform design system. 670+ components spanning iOS, Android, and web. 103 custom icons. Design tokens managed in Figma and synced to GitHub. Storybook documentation for engineering self-service. 37 organized Figma pages covering primitives, patterns, templates, and platform-specific adaptations.

The system was designed for engineer self-service. Component documentation included not just visual specs but behavior descriptions, state management guidelines, and accessibility requirements. An engineer could pick up a component from Storybook and implement it without needing a designer in the room.

Token architecture followed a three-tier model: primitive tokens (raw values), semantic tokens (contextual meaning), and component tokens (specific to individual components). This gave the system both flexibility and specificity.

THE IMPACT

670+ Components
37 Figma pages
103 Custom icons
3 Platforms

Techtonic transformed ZAR's development workflow. Engineers stopped waiting for designers to hand off specs — they could self-serve from Storybook. Design reviews shifted from "does this match the design?" to "does this solve the user problem?" — a much more valuable conversation.

The three-time rebuild was actually an advantage. Each version incorporated lessons from the last, and the final system was battle-tested against real product changes. It wasn't theoretical — it was proven to survive pivots.

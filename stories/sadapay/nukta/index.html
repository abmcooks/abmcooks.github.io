NUKTA — PAKISTAN'S FIRST DESIGN TOKEN SYSTEM

Building a token-based design system in 2021, before it was trendy. How Nukta gave SadaPay consistency at scale and created a shared language between design and engineering.

Company SadaPay Year 2021 – 2022 Role Head of Product Design

THE CHALLENGE

SadaPay was growing fast. What started as a solo designer's work had expanded to a team of eight, with multiple product squads shipping features in parallel. The problem was visible everywhere: inconsistent spacing, mismatched colors, buttons that looked different on every screen. Designers and engineers were speaking different languages.

Every new hire meant more divergence. Every feature shipped without a system meant more design debt. The Figma files were becoming unwieldy, and handoff to engineering was a constant source of friction. Components existed, but they weren't systematic — they were patches on a growing problem.

SadaPay needed a design system. Not just a component library, but a foundational system with tokens that could bridge the gap between design and code.

FIRST token-based design system in Pakistan's tech industry

THE APPROACH

I started with an audit. Every screen, every component, every color value, every spacing unit across the entire product. The audit revealed what I suspected: dozens of near-identical grays, inconsistent type scales, padding values that varied by screen, and button variants that served the same purpose but looked slightly different.

The approach was token-first. Before designing a single component, I defined the primitive tokens — colors, spacing, typography, elevation, border radii. Then I mapped those primitives to semantic tokens that described intent, not appearance. "color-surface-primary" instead of "gray-900". This meant the system could adapt (light mode, dark mode, new brand colors) without touching individual components.

I worked closely with engineering from day one. The token structure was designed to map directly to code variables. Every token in Figma had a corresponding token in the codebase. No translation layer, no interpretation — the same language in both tools.

Documentation was built alongside the system, not after. Every component had usage guidelines, do/don't examples, and accessibility notes. I wanted any new designer or engineer to be able to use Nukta without asking someone how it works.

THE SOLUTION

Nukta — named after the Urdu word for "dot" or "point" — became SadaPay's complete design system. It included primitive and semantic design tokens, a comprehensive component library in Figma with auto-layout and variants, coded components mirroring the Figma library, and thorough documentation covering usage, accessibility, and contribution guidelines.

The system covered every surface of the product: mobile (iOS and Android), web dashboard, internal tools, and marketing. Components were built with flexibility in mind — props and variants that handled edge cases without breaking the system's rules.

Nukta wasn't just a library. It was a contract between design and engineering — a shared source of truth that both teams could reference, contribute to, and rely on.

THE IMPACT

1st Token system in Pakistan
8 Designers using Nukta
3x Faster design-to-dev
0 Inconsistency debt

Nukta gave SadaPay consistency at scale during a period of rapid growth. New designers onboarded faster because the system taught them SadaPay's design language. Engineers stopped guessing at spacing and color values. Handoff became a non-issue — both sides spoke the same token language.

Perhaps most importantly, Nukta survived designer turnover. When team members left and new ones joined, the system held. The product stayed consistent because the knowledge wasn't in people's heads — it was in the system.
